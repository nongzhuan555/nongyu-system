# SDD (规格驱动开发) 指南

规格驱动开发的核心是“规格即契约”。在编写任何实现代码之前，必须先定义清晰、可校验的规格说明。

## 何时应用

### 必须使用
- **定义新接口**：在编写前后端逻辑前，必须先定义 OpenAPI/JSON Schema 规格。
- **设计复杂逻辑**：在实现算法前，必须先定义输入输出规格和状态转换图。
- **重构核心模型**：在修改数据库或核心数据结构前，必须先更新规格定义。

### 推荐使用
- 任何涉及多个组件协作的场景，用于消除集成歧义。
- 需要自动化生成测试用例或 Mock 数据时。

---

## SDD 核心原则

1. **规格先行 (Spec-First)**：先写规格文档，后写业务逻辑。
2. **契约即真理 (Spec as Truth)**：规格是项目唯一的真实来源，实现必须 100% 符合规格。
3. **可校验性 (Verifiability)**：规格必须是机器可读的，以便进行自动化契约测试。
4. **关注 WHAT 而非 HOW**：规格描述系统“做什么”和“约束是什么”，而不涉及具体实现细节。

---

## SDD 开发工作流

### 步骤 1: 书写或读取已有的PRD
- [ ] 当被要求开发某部分功能时，先检查是否有相关的PRD，若没有，则步骤卡住，要求用户提供一份相关PRD
- [ ] 读取PRD，理解功能需求和边界条件

### 步骤 2: 根据PRD编写Spec文档（AI写Spec）
- [ ] 根据PRD，编写规格文档，要求为markdown文件，内容包含以下部分：
    - 背景（why和what，这个需求是什么？为什么要做这个需求？）
	- 目标（需求要达到的目标和指标是什么？）
	- 边界（非目标，明确本次不做什么，缩小粒度，避免一次代码变更的影响范围太大）
	- 详细需求（完善PRD）
	- 业务流程
	- 验收标准和测试方案
- [ ] AI先审查规格文档，确保所有需求都被覆盖，没有遗漏或错误
- [ ] 要求用户对规格文档进行审查，确认无误后，再继续下一步骤

### 步骤 3: 基于Spec文档编写技术方案文档（AI写技术方案）
- [ ] 根据规格文档，编写技术方案文档，要求为markdown文件，内容包含以下部分：
    - 技术方案（包括数据库设计、API 设计、前端设计等）
	- 实现步骤（包括代码实现、测试用例、部署等）
	- 注意事项（如性能优化、安全考虑等）
- [ ] AI先审查技术方案文档，确保所有需求都被覆盖，没有遗漏或错误
- [ ] 要求用户对技术方案文档进行审查，确认无误后，再继续下一步骤

### 步骤 4: 基于技术方案和Spec文档编写实施计划（AI写实施计划）
- [ ] 根据技术方案文档和规格文档，编写实施计划，要求为markdown文件，内容包含以下部分：
    - 实施计划（包括时间表、资源分配、风险评估等）
	- 实施步骤（包括代码实现、测试用例、部署等）
	- 注意事项（如性能优化、安全考虑等）
- [ ] AI先审查实施计划文档，确保所有需求都被覆盖，没有遗漏或错误
- [ ] 要求用户对实施计划文档进行审查，确认无误后，再继续下一步骤

### 步骤 5: 按照实施计划进行实际编码（AI写代码）

### 步骤 6: 人工Review实现代码（可审查代码或实现效果）
---

## 决策准则

1. **规格是否足够严谨？**（是否存在模糊不清的字段或逻辑？）
2. **是否可以通过规格自动生成测试？**
3. **规格变更是否同步到了所有依赖方？**
